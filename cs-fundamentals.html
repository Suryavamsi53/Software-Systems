Core Computer Science Concepts

Operating Systems, Networks, Databases, LLD, and HLD for SDE Interviews

üñ•Ô∏è Operating Systems ‚Äì From Basics to Advanced
1. OS Fundamentals (Must Know) ‚ñº
  • What is an OS? Software that manages hardware/software resources and provides services for programs.
  • Goals: Performance (efficiency), Fairness (resource distribution), Isolation (process protection), Security.
  • Types: Batch (jobs in groups), Time-sharing (multitasking), Distributed (networked), Real-time (strict timing).
  • Kernel vs User Space: Kernel has full hardware access; User space is restricted.
  • System Calls: Interface for user programs to request kernel services (e.g., file I/O).
Interview Expectation
Explain OS as a resource manager that abstracts hardware complexity.
2. Processes & Threads (Top Topic) ‚ñº
  • Process: Program in execution. Heavyweight, separate memory, requires IPC.
  • Thread: Unit of execution within a process. Lightweight, shared memory, direct access.
  • PCB/TCB: Data structures storing state (Process/Thread Control Block).
  • States: New, Ready, Running, Waiting, Terminated.
  • Context Switching: Saving/restoring state to switch execution.

3. CPU Scheduling ‚ñº
  • Goal: Maximize throughput, minimize wait/response time.
  • Algorithms:
    • FCFS: First Come First Serve (simple, convoy effect).
    • SJF: Shortest Job First (optimal avg wait, starvation risk).
    • Round Robin: Time slices (fair, good for time-sharing).
    • Priority: Based on importance (starvation risk).
    • Multilevel Queue: Separate queues for different process types.
  • Issues: Starvation (low priority never runs), Aging (solution to starvation).
4. Context Switching ‚ñº

Definition: Storing state of active process/thread and loading another.

Cost: Expensive due to register saving, cache flushing, and TLB invalidation.

Follow-up
Why do too many threads hurt performance? Excessive context switching overhead consumes CPU cycles that could be used for actual work.
5. Inter-Process Communication (IPC) ‚ñº
  • Mechanisms: Pipes (unidirectional), Message Queues, Shared Memory (fastest), Sockets (network), Signals.
  • Comparison: Shared Memory is faster but needs sync; Message Passing is safer/easier for distributed systems.
6. Synchronization & Concurrency ‚ñº
  • Race Condition: Outcome depends on execution order.
  • Critical Section: Code accessing shared resources.
  • Solutions:
    • Mutex: Locking mechanism (binary).
    • Semaphore: Signaling mechanism (counter).
    • Monitor: High-level abstraction (e.g., Java synchronized).
  • Problems: Producer-Consumer, Readers-Writers, Dining Philosophers.
7. Deadlocks (Favorite Topic) ‚ñº

Definition: Processes waiting indefinitely for each other.

4 Conditions (Coffman): Mutual Exclusion, Hold and Wait, No Preemption, Circular Wait.

Handling: Prevention (break one condition), Avoidance (Banker's Algorithm), Detection & Recovery.

8. Memory Management ‚ñº
  • Logical vs Physical: CPU sees logical, MMU maps to physical.
  • Paging: Fixed-size blocks (pages). Eliminates external fragmentation.
  • Segmentation: Variable-size blocks (segments). Logical grouping.
  • Virtual Memory: Illusion of larger memory using disk (swap).
  • Thrashing: Excessive paging causing CPU to wait for I/O.
  • Replacement Algos: FIFO, LRU (Least Recently Used), Optimal.
9. Cache & Memory Hierarchy ‚ñº

Registers > L1/L2/L3 Cache > RAM > Disk.

Locality of Reference:

  • Temporal: Recently accessed items likely accessed again.
  • Spatial: Items near accessed ones likely accessed soon.
10. File Systems ‚ñº
  • Concepts: Files, Directories, Permissions.
  • Allocation: Contiguous (fast, fragmentation), Linked (no frag, slow random), Indexed (inode).
  • Inodes: Data structure storing file metadata.
  • Journaling: Logs changes before committing for crash recovery.
11. Disk Scheduling ‚ñº

Optimizing I/O seek time.

  • FCFS: Simple, fair.
  • SSTF: Shortest Seek Time First (fast, starvation risk).
  • SCAN/Elevator: Moves one way servicing requests, then reverses.
12. I/O Systems ‚ñº
  • DMA (Direct Memory Access): Device transfers data to RAM without CPU intervention.
  • Buffering: Smoothing speed mismatch between devices.
  • Spooling: Queueing jobs (e.g., printer).
13. Multi-Processor Systems ‚ñº
  • SMP: Symmetric Multiprocessing (shared memory, equal peers).
  • NUMA: Non-Uniform Memory Access (memory access time depends on location).
  • Affinity: Keeping a process on the same CPU for cache benefits.
14. Virtualization & Containers ‚ñº
  • VMs: Full OS virtualization via Hypervisor (Type 1 bare metal, Type 2 hosted).
  • Containers: OS-level virtualization (shared kernel). Lightweight.
  • Namespaces: Isolation (PID, Network).
  • Cgroups: Resource limiting (CPU, Memory).
15. Security & Protection ‚ñº
  • Protection: User vs Kernel mode (Ring 0/3).
  • Access Control: Authentication (Who), Authorization (What).
  • Isolation: Processes cannot access each other's memory.
16. Real-Time OS (RTOS) ‚ñº
  • Hard RTOS: Strict deadlines (miss = failure).
  • Soft RTOS: Deadlines preferred but flexible.
  • Priority Inversion: Low priority holds lock needed by high priority.
17. OS Performance & Tuning ‚ñº
  • Metrics: Throughput, Latency, Utilization.
  • Load Average: Avg number of processes running/waiting.
  • Bottlenecks: CPU bound vs I/O bound.
18. OS + Real-World Connections ‚ñº
  • Web Servers: Nginx uses event loops (epoll) to handle thousands of connections.
  • Databases: Use O_DIRECT to bypass OS cache for data integrity.
  • Containers: Docker relies on Linux Namespaces and Cgroups.
19. Common Interview Traps ‚ñº
Avoid These Mistakes
  • ‚ùå "Process and thread are same" (Threads share memory).
  • ‚ùå "More threads = faster" (Context switching kills perf).
  • ‚ùå "Deadlock always bad" (Sometimes ignored, e.g., Ostrich algorithm).
  • ‚ùå "Paging removes fragmentation completely" (Removes external, not internal).
  • ‚ùå "Context switching is free" (It's expensive).
20. Final Readiness Check ‚ñº
You are ready if you can:
  • Explain process vs thread without notes.
  • Solve a deadlock problem (Banker's algo).
  • Explain paging & page faults.
  • Describe Round Robin scheduling.
  • Connect OS concepts to backend systems (DBs, Web Servers).
üåê Computer Networks ‚Äì From Basics to Advanced
1. Network Fundamentals (Must Know) ‚ñº
  • Definition: Interconnected computers sharing resources.
  • Goals: Communication, Reliability, Scalability, Performance.
  • Types: LAN (Local), WAN (Wide), MAN (Metro), PAN (Personal).
  • Topologies: Bus, Star (common), Ring, Mesh (redundant).
  • Metrics: Bandwidth (capacity), Throughput (actual speed), Latency (delay).
  • Switching: Packet (internet) vs Circuit (old phone).

3. Physical & Data Link Layers ‚ñº
  • MAC Address: Physical hardware address (burned in).
  • ARP: Maps IP address to MAC address.
  • Ethernet: Standard for wired LANs.
  • Switch vs Hub: Switch is smart (sends to specific port), Hub broadcasts to all.
4. Network Layer (IP & Routing) ‚ñº
  • IP Addressing: IPv4 (32-bit) vs IPv6 (128-bit).
  • Subnetting: Dividing a network into smaller networks (CIDR notation).
  • NAT: Network Address Translation (Private IP -> Public IP).
  • Routing Protocols: BGP (Internet backbone), OSPF (Internal).
  • ICMP: Used for diagnostics (Ping, Traceroute).
5. Transport Layer (TCP vs UDP) ‚ñº

TCP (Transmission Control Protocol)

Connection-oriented, Reliable, Ordered, Flow Control. Used for Web, Email.

UDP (User Datagram Protocol)

Connectionless, Unreliable, Unordered, Fast. Used for Video, Gaming, DNS.

6. TCP Internals (Advanced) ‚ñº
  • 3-Way Handshake: SYN -> SYN-ACK -> ACK (Establish connection).
  • Flow Control: Sliding Window (Receiver tells Sender how much it can handle).
  • Congestion Control: Slow Start, Congestion Avoidance (Network health).
  • Reliability: Sequence numbers, ACKs, Retransmission.
7. Application Layer Protocols ‚ñº
  • HTTP/HTTPS: Stateless, Methods (GET, POST), Status Codes (200, 404, 500).
  • DNS: Domain Name System. Recursive vs Iterative queries. Caching is key.
  • SMTP/IMAP: Email protocols.
  • WebSockets: Persistent, full-duplex communication.
8. Security in Networks ‚ñº
  • TLS/SSL: Encryption for HTTPS. Handshake involves certificates.
  • Symmetric vs Asymmetric: Shared key vs Public/Private key pair.
  • Firewall: Filters traffic based on rules.
  • VPN: Encrypted tunnel over public network.
  • DDoS: Distributed Denial of Service attacks.
9. Performance & Reliability ‚ñº
  • Latency: Propagation + Transmission + Processing + Queuing delay.
  • Jitter: Variation in packet delay.
  • Packet Loss: Data failing to reach destination.
  • QoS: Prioritizing traffic (e.g., Voice over Download).
10. Load Balancing & Scaling ‚ñº
  • L4 vs L7: Transport (TCP/IP ports) vs Application (HTTP headers/URL).
  • Algorithms: Round Robin, Least Connections, Consistent Hashing.
  • Reverse Proxy: Server-side proxy (Nginx) for load balancing, caching, security.
11. Content Delivery Network (CDN) ‚ñº

Concept: Distributed network of edge servers to deliver content faster.

Usage: Caching static assets (images, CSS, JS) closer to user. Reduces latency and server load.

12. Socket Programming ‚ñº
  • Socket: Endpoint for communication (IP + Port).
  • Client-Server: Server listens on port, Client connects.
  • Ports: 0-1023 (Well-known), 1024-49151 (Registered).
13. Network Devices ‚ñº
  • Router: Connects different networks (L3). Uses IP.
  • Switch: Connects devices in same network (L2). Uses MAC.
  • Gateway: Entry/Exit point of a network.
  • Proxy: Intermediary for requests.
14. Network Virtualization & Cloud ‚ñº
  • VPC: Virtual Private Cloud (Isolated network in cloud).
  • Subnets: Public (Internet access) vs Private (Internal only).
  • Security Groups: Virtual firewall for instances.
  • NAT Gateway: Allows private subnet to access internet.
15. Real-World Scenarios ‚ñº
Scenario 1: Opening a Website
  1. DNS: Browser resolves domain to IP.
  2. TCP: 3-way handshake to establish connection.
  3. TLS: Handshake for encryption (HTTPS).
  4. HTTP: Browser sends GET request.
  5. Server: Processes and sends HTML response.
Scenario 2: Video Streaming

Uses CDN to serve content from edge. Uses Adaptive Bitrate to adjust quality based on bandwidth. Often uses TCP (DASH/HLS) for reliability, but live streaming might use UDP-based protocols.

üóÑÔ∏è Databases ‚Äì From Basics to Advanced
1. Database Fundamentals ‚ñº
  • DBMS: Software to store and manage data.
  • Relational (SQL): Structured, Schema-based, Tables (MySQL, PostgreSQL).
  • Non-Relational (NoSQL): Flexible, Schema-less, Documents/Key-Value (MongoDB, Redis).
  • SQL vs NoSQL: SQL for complex queries/transactions; NoSQL for scale/flexibility.
2. ACID Properties (Transactions) ‚ñº

The Guarantee of Reliability

  • Atomicity: All or nothing. If one part fails, the whole transaction rolls back.
  • Consistency: DB goes from one valid state to another (constraints enforced).
  • Isolation: Concurrent transactions don't interfere (Read Committed, Serializable).
  • Durability: Saved data survives power loss (Write-Ahead Log).
3. Indexing (Performance) ‚ñº

What is it? A data structure (like a B-Tree) that improves the speed of data retrieval operations on a database table.

Trade-off: Faster reads (SELECT), slower writes (INSERT/UPDATE) because the index must be updated.

  • B-Tree Index: Good for range queries (<, >, BETWEEN). Default in most DBs.
  • Hash Index: Good for equality checks (=). O(1) lookup.
  • Clustered Index: Sorts the data rows on disk (Primary Key). Only one per table.
  • Non-Clustered Index: Separate structure pointing to data rows. Can have multiple.
4. CAP Theorem (Distributed Systems) ‚ñº

In a distributed data store, you can only guarantee 2 out of 3:

  • Consistency (C): Every read receives the most recent write or an error.
  • Availability (A): Every request receives a response (no error), without guarantee of latest data.
  • Partition Tolerance (P): System continues to operate despite network message drops.
Real World
Since network partitions (P) are inevitable, you must choose between CP (Banking) and AP (Social Media Feed).
5. Scaling Databases ‚ñº
  • Vertical Scaling: Bigger machine (CPU/RAM). Limit: Hardware cost/max specs.
  • Horizontal Scaling: More machines.
  • Replication: Master-Slave (Read Replicas). Master handles writes, Slaves handle reads.
  • Sharding: Partitioning data across servers (e.g., Users A-M on DB1, N-Z on DB2).
6. Normalization ‚ñº

Organizing data to reduce redundancy.

  • 1NF: Atomic values (no lists in cells).
  • 2NF: No partial dependencies (all columns depend on full primary key).
  • 3NF: No transitive dependencies (columns depend ONLY on primary key).
  • Denormalization: Adding redundancy for read performance (Star Schema).
7. Real-World Examples ‚ñº
Example 1: Banking System (ACID)

Requirement: Transfer $100 from Alice to Bob.

Tech: RDBMS (PostgreSQL/MySQL).

Why: Need strong consistency (C) and Atomicity. If system crashes after debiting Alice but before crediting Bob, transaction must rollback.

Example 2: User Session Cache

Requirement: Store login tokens for millions of users with low latency.

Tech: Key-Value Store (Redis).

Why: Simple lookup (get token by ID), extremely fast (in-memory), TTL support (auto-expire sessions).

üìê Low Level Design (LLD) ‚Äì Object Oriented Design
1. LLD Fundamentals (OOP & UML) ‚ñº
  • Encapsulation: Bundling data and methods, hiding internal state (private fields).
  • Abstraction: Hiding complex implementation details, exposing only necessary interfaces.
  • Inheritance: Mechanism where a new class derives properties from an existing class.
  • Polymorphism: Ability of different classes to be treated as instances of the same general class (Overriding/Overloading).
  • UML Relationships:
    • Association: "Uses a" (Teacher uses Board).
    • Aggregation: "Has a" (Department has Teachers). Weak bond.
    • Composition: "Part of" (House has Rooms). Strong bond.
2. SOLID Principles (Deep Dive) ‚ñº

S - Single Responsibility Principle (SRP)

A class should have only one reason to change. Example: A `User` class should not handle DB operations; delegate that to `UserRepository`.

O - Open/Closed Principle (OCP)

Open for extension, closed for modification. Use Interfaces/Abstract classes so you can add new behavior without changing existing code.

L - Liskov Substitution Principle (LSP)

Subtypes must be substitutable for their base types. If `Square` inherits `Rectangle`, setting width shouldn't break logic expecting a rectangle.

I - Interface Segregation Principle (ISP)

Clients should not be forced to depend on interfaces they do not use. Split large interfaces into smaller, specific ones.

D - Dependency Inversion Principle (DIP)

High-level modules should not depend on low-level modules. Both should depend on abstractions (Interfaces). Use Dependency Injection.

3. Design Patterns (Interview Core) ‚ñº
  • Singleton: Ensure a class has only one instance (e.g., DB Connection). Watch out for thread safety!
  • Factory: Create objects without specifying the exact class (e.g., `ShapeFactory` returns `Circle` or `Square`).
  • Builder: Construct complex objects step by step (e.g., `HttpRequest.Builder().setUrl().setMethod().build()`).
  • Adapter: Allow incompatible interfaces to work together (e.g., XML to JSON adapter).
  • Observer: Notify subscribers of state changes (e.g., Event Listeners, Pub/Sub).
  • Strategy: Select algorithm at runtime (e.g., `PaymentStrategy`: CreditCard vs PayPal).
  • Decorator: Add behavior dynamically (e.g., `GzipStream(FileStream)`).
4. Concurrency in LLD ‚ñº

For 5+ YOE, you must handle thread safety.

  • Race Conditions: Multiple threads modifying shared data.
  • Locks/Mutex: Protecting critical sections.
  • Atomic Operations: Lock-free thread safety for simple counters.
  • Thread Pool: Reusing threads to avoid overhead.
5. Real-World Example: Parking Lot ‚ñº
Problem: Design a Parking Lot

Requirements: Multiple floors, multiple vehicle types (Car, Bike, Truck), hourly pricing, capacity checks.

Key Classes:

  • ParkingLot (Singleton): Manages levels.
  • Level: Contains list of spots.
  • ParkingSpot: Has type (Compact, Large), status (Occupied/Free).
  • Vehicle (Abstract): Car, Bike, Truck inherit from this.
  • Ticket: Entry time, Spot ID.

Design Patterns:

  • Factory: To create vehicle objects from input.
  • Strategy: PricingStrategy (Weekday vs Weekend).

Concurrency: bookSpot() must be synchronized to prevent double booking.

6. Real-World Example: Rate Limiter ‚ñº
Problem: Design a Rate Limiter Library

Requirements: Limit user to X requests per Y seconds.

Algorithms:

  • Token Bucket: Tokens added at rate r. Request consumes token. Good for bursty traffic.
  • Leaky Bucket: Requests processed at constant rate. Good for smoothing traffic.
  • Sliding Window Counter: Accurate but memory heavy.

Class Structure: RateLimiter interface, TokenBucket implementation, UserBucketMap.

üèóÔ∏è High Level Design (HLD) ‚Äì Scalable Architectures
1. Scalability Fundamentals ‚ñº
  • Vertical Scaling (Scale Up): Bigger server (RAM/CPU). Limited cap, single point of failure.
  • Horizontal Scaling (Scale Out): More servers. Infinite scale, requires load balancing.
  • Latency vs Throughput: Latency = time for one request. Throughput = requests per second.
  • Availability vs Consistency (CAP): In partition, choose A (always respond) or C (correct data).
2. Load Balancing & Caching ‚ñº

Load Balancing

Distributes traffic across servers. Algorithms: Round Robin, Least Connections, Consistent Hashing (crucial for distributed caches).

Caching Strategies

  • Cache-Aside: App checks cache -> DB -> updates cache. (Most common).
  • Write-Through: App writes to cache -> cache writes to DB. (Data consistency).
  • Eviction Policies: LRU (Least Recently Used), LFU (Least Frequently Used).
3. Database Scaling ‚ñº
  • Replication: Master-Slave. Master for writes, Slaves for reads. Async replication lag issues.
  • Sharding: Partitioning data (e.g., UserID % 10). Complex joins, re-balancing is hard.
  • NoSQL Types: Key-Value (Redis), Document (MongoDB), Columnar (Cassandra), Graph (Neo4j).
4. Communication & Async Processing ‚ñº
  • REST: Resource-based, HTTP, JSON. Standard.
  • gRPC: Action-based, Protobuf, HTTP/2. High performance, internal microservices.
  • WebSockets: Bi-directional, persistent. Chat apps, live feeds.
  • Message Queues: Kafka (Log-based, high throughput), RabbitMQ (Queue-based, complex routing). Decouples producer/consumer.
5. Real-World Example: URL Shortener ‚ñº
Design TinyURL

Requirements: 100M new URLs/month. Read:Write = 100:1. Short alias (7 chars).

API: create(long_url) -> short_url, get(short_url) -> long_url.

DB Schema: {id, long_url, short_url, created_at}. Indexed on short_url.

Algorithm: Base62 encoding of a unique ID. Use a Key Generation Service (KGS) to pre-generate unique tokens to avoid collision checks.

Scaling: Cache heavily (Cache-Aside). 20% of URLs generate 80% of traffic.

6. Real-World Example: Chat App ‚ñº
Design WhatsApp/Slack

Requirements: 1:1 Chat, Group Chat, Online Status, Persistent History.

Protocol: WebSockets for real-time delivery. HTTP for media upload.

Database: Cassandra/HBase (Wide-column) for chat history (Write heavy, time-series access). SQL for User Profiles.

Status Service: Heartbeat mechanism. Redis for "Last Seen".
