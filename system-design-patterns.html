Common strategies for Load Balancing, Caching, and Scaling

‚öñÔ∏è Load Balancing Algorithms

How to distribute traffic across healthy servers.

Round Robin

Requests are distributed sequentially (Server 1, then 2, then 3, back to 1).
Pros: Simple, fair if servers are equal.
Cons: Doesn't account for server load or capacity.

Least Connections

Send request to the server with fewest active connections.
Pros: Good for long-lived connections (WebSocket, DB).
Cons: Requires tracking state.

Consistent Hashing

Hash the request ID (e.g., UserID) to map to a server. Used in distributed caches.
Pros: Adding/removing servers only affects k/N keys.
Cons: Complex to implement (virtual nodes).

‚ö° Caching Strategies

Where to place the cache and how to update it.

1. Cache-Aside (Lazy Loading)
  • App checks Cache. If miss, App reads DB, then writes to Cache.
  • Best for: Read-heavy workloads. Data is loaded only when requested.
  • Risk: Cache stampede (thundering herd) if many users request same missing key.
2. Write-Through
  • App writes to Cache, and Cache writes to DB synchronously.
  • Best for: Data consistency. Cache is always up-to-date.
  • Risk: Higher write latency (two writes).
3. Write-Back (Write-Behind)
  • App writes to Cache only. Cache writes to DB asynchronously later.
  • Best for: Write-heavy workloads. Low latency.
  • Risk: Data loss if cache crashes before syncing to DB.
üóÑÔ∏è Database Scaling Patterns
Replication (Master-Slave)

One Master (Writes), Multiple Slaves (Reads).
Use when: Read-heavy application (e.g., Social Media feed).
Issue: Replication lag (Eventual Consistency).

Sharding (Partitioning)

Split data across multiple DB servers based on a Shard Key (e.g., UserID % 4).
Use when: Data volume exceeds single server capacity.
Issue: Joins across shards are expensive/impossible.

For Amazon interviews, focus on Consistent Hashing (DynamoDB) and Cache-Aside (ElastiCache).
