Master complexity analysis for Amazon coding interviews

üìö What is Complexity Analysis?

Complexity analysis measures how the algorithm's resource usage (time/memory) grows as input size increases.

Why it matters for Amazon interviews:
  • Shows you understand algorithm efficiency
  • Helps you choose between multiple solutions
  • Demonstrates scalability thinking (core Amazon value)
  • Essential for system design later in interviews
  • Proves you can handle large-scale problems

Two main metrics:

  • Time Complexity: How many operations the algorithm performs
  • Space Complexity: How much extra memory the algorithm uses
‚è±Ô∏è Time Complexity
Big-O Complexity Chart
O(1) - Constant Time EXCELLENT

Definition: Takes the same time regardless of input size. Operation count doesn't scale.

Examples:
  • Array/list access by index
  • Hash table lookup (average)
  • Set membership check
  • Basic arithmetic operations
Code Example: 
def get_first(arr):
    return arr[0]  # Always 1 operation

def lookup_hash(hash_map, key):
    return hash_map[key]  # O(1) average
O(log n) - Logarithmic Time EXCELLENT

Definition: Time grows logarithmically with input size. Problem space is cut in half each iteration.

Examples:
  • Binary search (sorted array)
  • Balanced binary tree operations
  • Heap operations (insert/delete)
  • Divide and conquer algorithms
Code Example: 
def binary_search(arr, target):
    left, right = 0, len(arr) - 1
    while left <= right:
        mid = (left + right) // 2
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            left = mid + 1  # Cut search space
        else:
            right = mid - 1
    return -1
üí° Insight: For n=1,000,000, O(log n) ‚âà 20 operations. Incredibly efficient!
O(n) - Linear Time GOOD

Definition: Time grows proportionally with input size. Single loop through data.

Examples:
  • Simple array iteration
  • Linear search (find in unsorted)
  • Finding min/max element
  • Sliding window with single pass
Code Example: 
def find_max(arr):
    max_val = arr[0]
    for num in arr:  # Iterate through all
        max_val = max(max_val, num)
    return max_val

def linear_search(arr, target):
    for i in range(len(arr)):  # O(n)
        if arr[i] == target:
            return i
    return -1
üí° Insight: Often acceptable for n up to 10^8. Baseline for many problems.
O(n log n) - Linearithmic Time GOOD

Definition: Combination of linear and logarithmic. Typical for efficient sorting algorithms.

Examples:
  • Merge sort
  • Quick sort (average)
  • Heap sort
  • Sorting + linear operations
Code Example: 
def merge_sort(arr):
    if len(arr) <= 1:
        return arr
    
    mid = len(arr) // 2
    left = merge_sort(arr[:mid])    # O(log n) depth
    right = merge_sort(arr[mid:])   # O(log n) depth
    return merge(left, right)       # O(n) merge
    
# Total: O(n) * O(log n) = O(n log n)
üí° Insight: Default choice for sorting. Most sorting algorithms achieve this.
O(n¬≤) - Quadratic Time ACCEPTABLE

Definition: Time grows with square of input size. Nested loops.

Examples:
  • Bubble sort, insertion sort
  • Nested loops (all pairs)
  • Matrix operations
  • Brute force comparison
Code Example: 
def bubble_sort(arr):
    for i in range(len(arr)):      # O(n)
        for j in range(len(arr)-1):  # O(n)
            if arr[j] > arr[j+1]:
                arr[j], arr[j+1] = arr[j+1], arr[j]
    return arr
# Total: O(n) * O(n) = O(n¬≤)
‚ö†Ô∏è Caution: Works for n < 10,000. For n=100,000, takes billions of operations.
O(n¬≥) - Cubic Time USUALLY AVOID

Definition: Triple nested loops. Grows very quickly.

Examples:
  • 3D matrix traversal
  • Triple nested loops
  • Some DP solutions
Code Example: 
def triple_loop(n):
    count = 0
    for i in range(n):      # O(n)
        for j in range(n):    # O(n)
            for k in range(n):  # O(n)
                count += 1
    return count
# Total: O(n¬≥)
‚ùå Problem: For n=100, already 1 million operations. Only use if unavoidable.
O(2^n) - Exponential Time AVOID - OPTIMIZE!

Definition: Time doubles with each additional input. Extremely fast growth.

Examples:
  • Recursive backtracking (no memoization)
  • Generate all subsets
  • Brute force password cracking
Code Example: 
def fibonacci_slow(n):
    if n <= 1:
        return n
    return fibonacci_slow(n-1) + \
           fibonacci_slow(n-2)
# O(2^n) - recalculates same values
üö® Critical: For n=50, this is 10^15 operations! Use memoization or optimization.
O(n!) - Factorial Time IMPRACTICAL

Definition: Extremely rare. Used for generating all permutations.

Example: Generating all permutations of a list
‚ö†Ô∏è Never use unless there's absolutely no alternative.
For n=13, this is 6 billion operations. For n=20, impractical.
üìâ Amortized Analysis

Definition: The average time per operation over a sequence of operations. Even if one operation is slow, if most are fast, the amortized cost is low.

Example: Dynamic Array (ArrayList / Slice)

Operation: Appending to an array.
Most of the time: O(1) - just add element at end.
Occasionally: O(n) - array is full, allocate double size, copy all elements.
Amortized Cost: O(1).

Why? If we double capacity, we won't resize again for another n operations. The expensive O(n) copy is "paid for" by the n cheap O(1) insertions.

üíæ Space Complexity

Space complexity measures extra memory used by the algorithm (not counting input).

O(1) - Constant Space OPTIMAL

Definition: Uses constant memory regardless of input size. Only a few variables.

Examples:
  • In-place array sorting
  • Two pointer technique
  • Simple counter variables
Code Example: 
def reverse_array(arr):
    left, right = 0, len(arr) - 1
    while left < right:
        arr[left], arr[right] = \
            arr[right], arr[left]
        left += 1
        right -= 1
    return arr
# Only 2 variables: O(1) space
O(n) - Linear Space COMMON

Definition: Memory grows proportionally with input. Creating a new data structure.

Examples:
  • Hash map for storing all elements
  • Result array same size as input
  • Queue/stack for BFS/DFS
Code Example: 
def two_sum(nums, target):
    seen = {}  # O(n) space
    for num in nums:
        complement = target - num
        if complement in seen:
            return [seen[complement], 
                   nums.index(num)]
        seen[num] = nums.index(num)
    return []
O(log n) - Logarithmic Space GOOD

Definition: Recursion depth or partial data structures.

Examples:
  • Binary search (recursion depth)
  • Balanced tree operations
  • Divide and conquer
Code Example: 
def binary_search(arr, target):
    def search(left, right):
        if left > right:
            return -1
        mid = (left + right) // 2
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            return search(mid+1, right)
        else:
            return search(left, mid-1)
    # Recursion depth: O(log n)
    return search(0, len(arr)-1)
O(n¬≤) - Quadratic Space ACCEPTABLE

Definition: 2D matrix or nested data structures.

Examples:
  • 2D DP table
  • Adjacency matrix
  • Cartesian product results
Code Example: 
def create_dp_table(n, m):
    # 2D table: O(n*m) space
    dp = [[0 for _ in range(m)]
          for _ in range(n)]
    # Other calculations...
    return dp

üéØ When to Optimize Complexity
‚úÖ This Complexity is FINE
  • O(1): Always perfect
  • O(log n): For n up to 10^8
  • O(n): For n up to 10^8
  • O(n log n): For n up to 10^6
  • O(n¬≤): For n up to 10^4
‚ö†Ô∏è This Complexity NEEDS Optimization
  • O(n¬≤): For n > 10^5
  • O(n¬≥): Usually always
  • O(2^n): Always! (use DP)
  • O(n!): Only for n < 12
‚≠ê Interview Tips for Discussing Complexity
üí¨ Always Explain, Don't Just State

Don't say "O(n log n)". Say "O(n log n) because we sort the array and then iterate once."

üîÑ Discuss Trade-offs

"This uses O(n) space to achieve O(n) time. Without that space, it would be O(n¬≤) time."

üîç Count Operations Explicitly

For small examples, trace through and count: "This loop runs n times, and each iteration does 5 operations."

üìä Relate to Real Numbers

"For n=1 million, O(n) is 1 million operations. O(n¬≤) would be 1 trillion‚Äîdefinitely too slow."

üöÄ Suggest Optimizations

"We could optimize this to O(n) if we use a hash map instead, though it would cost O(n) space."
