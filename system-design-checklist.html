A 45-minute structured guide to ace your system design interview.

1. Define Functional Requirements 2 min

What exactly does the system do? (e.g., Users can post tweets, follow others, view timeline). Limit to top 3-4 core features.

2. Define Non-Functional Requirements 2 min

Scalability (DAU/MAU), Latency (p99), Availability (CAP theorem), Consistency (Strong vs Eventual), Durability.

3. Back-of-the-Envelope Estimations 3 min

Traffic (QPS), Storage (TB/year), Bandwidth (Mbps). Do this ONLY if relevant to design choices (e.g., sharding).

üìê Phase 2: High-Level Design (10-15 min)

4. API Design 3 min

Define endpoints. POST /tweet (user_id, content), GET /feed (user_id, cursor). Keep it REST/gRPC.

5. Database Schema 3 min

SQL vs NoSQL decision. Define tables/collections. Primary keys, Partition keys. Relationships (1:N, M:N).

6. High-Level Diagram 5 min

Draw boxes: Client -> LB -> Service -> DB. Add Cache, Queue, CDN where appropriate. Walk through the flow.

7. Scaling & Bottlenecks 5 min

Identify single points of failure. Discuss Sharding (DB), Replication (Read/Write), Load Balancing strategies.

8. Specific Component Drill-down 10 min

Interviewer's choice. E.g., "How to generate unique IDs?", "How to handle fan-out?", "How to secure data?".
