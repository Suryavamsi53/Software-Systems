Deep dive into real-world architectures: TinyURL & WhatsApp

üîó Design TinyURL (URL Shortener)
1. Requirements & Estimation ‚ñº

Functional: Shorten URL, Redirect, Custom alias (optional).

Non-Functional: Highly available, Low latency, Read-heavy.

Estimation:

  • Write: 100M URLs/month. Read: 10B/month (100:1 ratio).
  • Storage: 500 bytes/URL * 100M * 12 * 5 years = 30TB.
2. High Level Design ‚ñº
Client <--> Load Balancer <--> Web Servers <--> Cache (Redis) ^ | Database

Flow: Check Cache -> If miss, check DB -> Return Long URL.

3. Key Component: Hash Generation ‚ñº
Base62 Encoding

Characters: A-Z, a-z, 0-9 (62 chars).

Length 7: 62^7 = ~3.5 Trillion combinations (Enough).

Collision Handling: Use a Key Generation Service (KGS) to pre-generate unique tokens and store in a DB/Redis set. Web servers fetch tokens from KGS.

üí¨ Design WhatsApp (Chat App)
1. Requirements ‚ñº

Functional: 1-on-1 Chat, Group Chat, Sent/Delivered/Read Receipts, Online Status.

Non-Functional: Low latency (real-time), Consistency (messages ordered), High Availability.

2. Architecture & Protocols ‚ñº
WebSocket vs HTTP

Use WebSocket for persistent bi-directional connection. HTTP is too slow (polling) for real-time chat.

Gateway Service: Maintains WebSocket connections with users.

3. Database Schema ‚ñº

User Service: SQL (Relational) for profile data, friends list.

Chat Service: NoSQL (Cassandra/HBase) for chat history.

Table: Messages
----------------
message_id (Partition Key)
chat_id (Clustering Key)
sender_id
content
timestamp
status (sent/delivered/read)

Why NoSQL? Extremely high write throughput, simple key-value access patterns, horizontal scaling.
