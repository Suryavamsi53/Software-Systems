6Ô∏è‚É£ Arrays‚ñº

Types & Definition

Arrays are fixed-length sequences of items of the same type. The length is part of the type (e.g., [5]int is different from [10]int).

Implementation

Arrays are value types. Assigning one array to another copies all elements. Passing an array to a function copies it (expensive for large arrays).

Examples

// Declaration
var a [2]string             // ["" ""] (Zero values)
a[0] = "Hello"

primes := [6]int{2, 3, 5, 7, 11, 13} // Initialized
auto := [...]int{1, 2, 3}   // Compiler counts elements (3)
  • Fixed length, value type (copy on assignment).
7Ô∏è‚É£ Slices (Deep Dive)‚ñº

1. What it is

A slice is a dynamic, flexible view over an underlying array. Internally it has a pointer to the array, a length (visible elements), and a capacity (max elements without reallocation).

2. Why it exists

Arrays are fixed-size and copied on assignment. Slices solve this by providing dynamic growth, cheap passing (it's just a descriptor), and flexible memory usage.

3. When to use

  • List size is unknown.
  • Passing collections to functions (cheap).
  • Implementing stacks, queues, buffers.

4. When NOT to use

  • Fixed-size data is required (use Array).
  • Memory ownership must be explicit.
  • Concurrent writes are expected (not thread-safe).

5. Real-world use cases

HTTP request buffers, DB result sets, Job queues, In-memory caches.

Pros
  • Dynamic size
  • Cheap to pass
  • Fast iteration
  • Idiomatic Go
Cons
  • Hidden allocations
  • Capacity traps
  • Memory leaks via slicing large arrays

8. Interview Expectations

You MUST know: len vs cap, append reallocation mechanics, slice header behavior, and backing array sharing.

9. Common Mistakes

  • Returning pointer to slice (*[]int).
  • Assuming append modifies the original slice header in the caller.
  • Forgetting to copy when keeping a small part of a large array.

10. Clean vs Bad Code

// ‚ùå Bad: Returning pointer to slice
func get() *[]int {
    s := []int{1, 2}
    return &s
}

// ‚úÖ Clean: Return slice directly
func get() []int {
    s := make([]int, 0, 10)
    return append(s, 1, 2)
}
8Ô∏è‚É£ Maps (Deep Dive)‚ñº

1. What it is

A built-in hash table implementation. It maps keys to values. Keys must be comparable (supports ==).

2. Why it exists

To provide O(1) average time complexity for lookups, insertions, and deletions. Essential for associative data.

3. When to use

  • Fast lookups by ID/Key.
  • Counting frequencies (e.g., word count).
  • Caching results (memoization).
  • Deduplicating items (using map as a set).

4. When NOT to use

  • Order matters (Maps are unordered).
  • Sequential access is needed (Slices are faster).
  • Memory is extremely constrained (Maps have overhead).
  • Concurrent access without synchronization (Panic).

8. Interview Expectations

Know that maps are references. Writing to a nil map panics. Reading from a nil map returns zero value. Maps are not thread-safe (use sync.RWMutex or sync.Map).

10. Clean vs Bad Code

// ‚ùå Bad: Writing to nil map
var m map[string]int
m["key"] = 1 // PANIC!

// ‚úÖ Clean: Initialize with make
m := make(map[string]int)
m["key"] = 1

// ‚úÖ Clean: Check existence
if val, ok := m["key"]; ok {
    fmt.Println("Found:", val)
}

üîü Pointers‚ñº

Concept

Holds the memory address of a value. Used to share data or modify state.

Examples

i := 42
p := &i         // & generates a pointer to i
fmt.Println(*p) // * dereferences the pointer (reads i)
*p = 21         // sets i through the pointer

// No pointer arithmetic in Go (unlike C)
  • Address operator &, Dereferencing *.
  • Pass by value (pointers are values too).
1Ô∏è‚É£1Ô∏è‚É£ Methods‚ñº

Concept

Functions attached to a type. The receiver argument appears between func and the method name.

Examples

import "math"

// Value Receiver: Operates on a copy (cannot modify original)
func (v Vertex) Abs() float64 { return math.Sqrt(v.X*v.X + v.Y*v.Y) }

// Pointer Receiver: Operates on actual value (can modify)
func (v *Vertex) Scale(f float64) { v.X = v.X * f; v.Y = v.Y * f }
  • Value Receivers: Copy of struct.
  • Pointer Receivers: Modify original struct.
üü£ INTERFACES & TYPES
1Ô∏è‚É£2Ô∏è‚É£ Interfaces (Deep Dive)‚ñº

1. What it is

A set of method signatures. Interfaces define behavior, not data. Go interfaces are implemented implicitly (Duck Typing).

2. Why it exists

To decouple implementation from usage. Allows writing functions that accept any type satisfying a behavior (e.g., io.Reader).

3. When to use

  • Dependency Injection (swapping real DB for mock DB).
  • Generic algorithms (e.g., sort.Interface).
  • Defining boundaries between packages.

8. Interview Expectations

Explain implicit implementation. Know the "Nil Interface" trap (interface is nil only if type and value are nil). Understand empty interface interface{} (or any).

10. Clean vs Bad Code

// ‚ùå Bad: Returning concrete type as interface (Nil Trap risk)
func Get() io.Reader {
    var b *bytes.Buffer = nil
    return b // Returns (type=*bytes.Buffer, value=nil) != nil
}

// ‚úÖ Clean: Return nil explicitly
func Get() io.Reader {
    return nil
}

1Ô∏è‚É£5Ô∏è‚É£ Goroutines‚ñº

Concept

Lightweight threads managed by Go runtime. 2KB stack.

Examples

func main() {
    done := make(chan struct{})
    
    go func() {
        fmt.Println("world")
        close(done) // Signal completion
    }()
    
    fmt.Println("hello")
    <-done // Wait for goroutine
}
  • Lightweight threads (M:N scheduler).
  • Stack growth, Goroutine leaks.
1Ô∏è‚É£6Ô∏è‚É£ Channels (Deep Dive)‚ñº

1. What it is

A channel is a typed conduit used to communicate between goroutines. It enforces the philosophy: "Don't communicate by sharing memory; share memory by communicating."

2. When to use

  • Passing ownership of data between goroutines.
  • Coordinating/Signaling (e.g., "I'm done").
  • Implementing worker pools.

3. When NOT to use

  • Protecting simple counters (Use atomic or Mutex).
  • Low-latency critical sections (Channels have overhead).
Pros
  • Thread-safe
  • Clear ownership
  • Deadlock detection (runtime)
Cons
  • Overhead vs Mutex
  • Deadlocks if misused
  • Harder debugging

8. Interview Expectations

Explain buffered vs unbuffered, who closes the channel (sender), select statement, and channel vs mutex trade-offs.

10. Clean vs Bad Code

// ‚ùå Bad: Busy waiting
for !done {
    time.Sleep(100 * time.Millisecond)
}

// ‚úÖ Clean: Signal channel
done := make(chan struct{})
go func() { close(done) }()
<-done
1Ô∏è‚É£7Ô∏è‚É£ select Statement‚ñº

Concept

Waits on multiple channel operations. Like a switch for channels.

Examples

select {
case msg := <-c1:
    fmt.Println("Received", msg)
case <-time.After(1 * time.Second):
    fmt.Println("Timeout")
default:
    fmt.Println("Non-blocking check")
}
  • Multiplexing channels.
  • Timeouts, Non-blocking operations.
1Ô∏è‚É£8Ô∏è‚É£ sync Package‚ñº

Concept

Primitives for synchronization when channels aren't appropriate.

Examples

// Mutex (Locking)
var mu sync.Mutex
mu.Lock()
count++ // critical section
mu.Unlock()

// WaitGroup (Waiting for goroutines)
var wg sync.WaitGroup
wg.Add(1)
go func() { defer wg.Done(); ... }()
wg.Wait()
  • sync.Mutex, sync.RWMutex.
  • sync.WaitGroup, sync.Once.
  • Atomic operations.

2Ô∏è‚É£8Ô∏è‚É£ Context Package (Deep Dive)‚ñº

1. What it is

A standard package for carrying deadlines, cancellation signals, and request-scoped values across API boundaries and goroutines.

2. Why it exists

To prevent goroutine leaks and wasted work. If a user cancels a request, all downstream work (DB queries, API calls) should stop immediately.

3. When to use

  • Every long-running operation (DB, Network).
  • Passing Trace IDs or User Auth info.
  • Setting timeouts for external calls.

8. Interview Expectations

Always pass ctx as the first argument. Never store ctx in a struct (pass it explicitly). Use defer cancel() to avoid leaks.

10. Clean vs Bad Code

// ‚ùå Bad: Ignoring context
func dbCall() {
    time.Sleep(5 * time.Second) // Can't be cancelled
}

// ‚úÖ Clean: Respecting cancellation
func dbCall(ctx context.Context) error {
    select {
    case <-time.After(5 * time.Second):
        return nil
    case <-ctx.Done():
        return ctx.Err() // Cancelled
    }
}
