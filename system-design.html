üìò What is System Design?

System design is the process of defining the architecture, components, modules, interfaces, and data for a system to satisfy specified requirements. In Amazon interviews, this tests your ability to build scalable, reliable, and maintainable systems.

üß† Core Concepts
Scalability
  • Vertical Scaling (Scale Up): Adding more power (CPU, RAM) to an existing server. Easy but has a limit.
  • Horizontal Scaling (Scale Out): Adding more servers to the pool. Preferred for distributed systems.
CAP Theorem

In a distributed system, you can only have 2 of 3:

  • Consistency: Every read receives the most recent write or an error.
  • Availability: Every request receives a (non-error) response, without the guarantee that it contains the most recent write.
  • Partition Tolerance: The system continues to operate despite an arbitrary number of messages being dropped or delayed by the network between nodes.
üìê CAP Theorem Visualized
Pick Two: The Distributed System Trade-off

You cannot have all three simultaneously in a distributed system with network partitions.

CP (Consistency + Partition)

System waits or returns error if partition occurs. Data is always correct.

Ex: Banking, Inventory

AP (Availability + Partition)

System always responds, even with stale data. "Eventual Consistency".

Ex: Social Feed, Shopping Cart

CA (Consistency + Availability)

Not possible in distributed systems (network partitions are inevitable).

Ex: Single Node DB

üß± Key Components
Load Balancer (LB)

Distributes incoming network traffic across multiple servers. Ensures no single server bears too much load. Can be Hardware (F5) or Software (NGINX, HAProxy, AWS ELB).

Caching

Temporary storage for frequently accessed data. Reduces database load and latency.

  • CDN (Content Delivery Network): Caches static assets (images, CSS) closer to users (e.g., CloudFront).
  • Distributed Cache: Redis or Memcached for application data.
Database Sharding

Splitting a large database into smaller, faster, more easily managed parts called data shards. Horizontal scaling for databases.

üì¶ Amazon-Specific Technologies
Know These AWS Services
  • DynamoDB: NoSQL key-value database. High availability, scalability. Great for shopping carts, user preferences.
  • S3 (Simple Storage Service): Object storage. Infinite scaling. Used for images, logs, backups.
  • SQS (Simple Queue Service): Message queue for decoupling components. Asynchronous processing.
  • SNS (Simple Notification Service): Pub/Sub messaging for broadcasting events.
  • Lambda: Serverless compute. Run code without provisioning servers.
üìù Example: Design a URL Shortener (TinyURL)

1. Requirements

  • Functional: Given a long URL, return a unique short URL. Redirect short URL to long URL.
  • Non-Functional: Highly available, low latency, scalable.

2. Back-of-the-Envelope Estimation

Traffic: 100M new URLs/month.
Read:Write ratio: 10:1.
Storage: 100M * 500 bytes * 12 months * 5 years = ~30 TB.

3. High-Level Design

User -> Load Balancer -> Web Servers -> Cache (Redis) -> Database

4. Database Schema

Table: URLs
- Hash (PK): varchar(7)
- OriginalURL: varchar(512)
- CreationDate: datetime

5. Key Component: Generating the Hash

Use Base62 encoding (A-Z, a-z, 0-9).
62^7 = ~3.5 trillion combinations. Enough for years.
Use a distributed ID generator (like Snowflake or a dedicated Key Generation Service) to ensure unique IDs before encoding.
